## Status

**Proposed**

## Context

The repository ships a static NVIDIA `xorg.conf` file at
`src/aps/configs/nvidia/xorg_04-06-25.conf`. This file is a full X server
configuration generated by `nvidia-settings` for a specific GPU (GeForce RTX
2060) and a specific monitor (ASUS VG27A at 144 Hz). Deploying it as part of a
general-purpose setup tool creates several problems:

- **CTX-001**: **Hardware-specific assumptions** — The config hard-codes a
  monitor model, refresh rate range (48–144 Hz), and GPU board name. Applying it
  to a different GPU/monitor combination can cause X to fail to start, output at
  the wrong resolution, or ignore user display preferences.
- **CTX-002**: **Modern drivers auto-detect** — Current NVIDIA proprietary
  drivers (535+) and the open-source `nvidia-open` driver perform automatic
  monitor detection and mode-setting. A static `xorg.conf` overrides this
  auto-detection, which is the opposite of what most users want.
- **CTX-003**: **nvidia-settings is the correct tool** — NVIDIA officially
  recommends using `nvidia-settings` to generate per-machine X configuration.
  Users who need specific overrides (Coolbits, custom resolutions) should run
  `nvidia-settings --save` on their own system.
- **CTX-004**: **Maintenance burden** — The config file must be updated whenever
  the reference hardware changes, or it silently ships stale settings. It is
  already dated (`xorg_04-06-25.conf`) and tied to driver version 570.124.04.

The current `nvidia.py` module provides three useful functions — `setup_cuda()`,
`switch_to_open_driver()`, and `setup_vaapi()` — none of which reference the
bundled xorg.conf file. The file is effectively dead code from the module's
perspective.

## Decision

Remove the NVIDIA xorg.conf setup artefact from the repository and ensure no
code path deploys a static X configuration for NVIDIA GPUs.

### Scope of removal

1. **Delete** `src/aps/configs/nvidia/xorg_04-06-25.conf`.
2. **Delete** the `src/aps/configs/nvidia/` directory if no other files remain.
3. **Verify** that `src/aps/hardware/nvidia.py` does not reference the file or
   attempt to copy it to `/etc/X11/xorg.conf.d/`. Currently the module does not,
   so no code changes to `nvidia.py` are required.
4. **Update** `AGENTS.md` repository structure to remove the
   `nvidia/xorg_04-06-25.conf` entry.
5. **Add a note** to the NVIDIA setup component's help text or documentation
   advising users to run `nvidia-settings` for custom X configuration.

### Rationale

- The existing NVIDIA module's three features (CUDA, open driver, VA-API) are
  generic and distro-aware. They do not need or use a static xorg.conf.
- Removing it eliminates a footgun for users who might discover and manually
  deploy the file.
- Other hardware modules (Intel, touchpad) ship configs that are genuinely
  generic (e.g., `20-intel.conf` sets TearFree, `99-touchpad.conf` enables tap).
  The NVIDIA xorg.conf is fundamentally different: it is a full server layout
  specific to one machine.

## Consequences

### Positive

- **POS-001**: Eliminates the risk of users deploying an incompatible X
  configuration that could prevent X from starting.
- **POS-002**: Reduces maintenance burden — no need to keep the config file in
  sync with driver or hardware changes.
- **POS-003**: Aligns the NVIDIA workflow with upstream NVIDIA guidance
  (`nvidia-settings` for per-machine configuration).
- **POS-004**: Simplifies the `src/aps/configs/` tree by removing a
  single-file subdirectory.

### Negative

- **NEG-001**: Users who previously relied on the bundled config as a reference
  template lose that convenience. Mitigated by documenting `nvidia-settings
  --save` in the setup help text.
- **NEG-002**: If a future use case requires shipping generic NVIDIA xorg
  snippets (e.g., Coolbits-only fragment), the directory will need to be
  recreated.
- **NEG-003**: The removal is a breaking change for any workflow that explicitly
  copies this file (unlikely, since no code path references it).

## Alternatives Considered

### Keep the file but mark it as a template

- **ALT-001**: **Description**: Rename to `xorg.conf.example` and add comments
  warning that it must be customized per machine.
- **ALT-002**: **Rejection Reason**: Users who find template files in a config
  directory are likely to copy them as-is. The file is too specific to serve as
  a useful starting point — `nvidia-settings --save` produces a better template
  than any hand-maintained file.

### Replace with a minimal Coolbits-only snippet

- **ALT-003**: **Description**: Ship a 5-line xorg.conf.d snippet that only
  enables `Option "Coolbits" "4"` for fan/overclock control.
- **ALT-004**: **Rejection Reason**: Coolbits is a niche requirement. Adding it
  as an optional flag to the `nvidia` setup component (`aps setup nvidia
  --coolbits`) would be cleaner than shipping a static file.

### Move the file to `examples/`

- **ALT-005**: **Description**: Relocate the file outside the package tree into
  `examples/nvidia/` so it is not shipped with the installed package.
- **ALT-006**: **Rejection Reason**: Still implies the project endorses the
  config. Deleting it entirely sends a clearer message that per-machine X
  configuration should be generated locally.

## Implementation Notes

- **IMP-001**: After deletion, run the full test suite (`uv run pytest -m "not
  slow"`) to confirm no tests reference the file.
- **IMP-002**: Update the `AGENTS.md` repository structure section to remove the
  `nvidia/` subtree from the configs listing.
- **IMP-003**: Consider adding a `logger.info` message in `nvidia.configure()`
  directing users to `nvidia-settings` for X configuration needs.
